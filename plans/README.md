## Original findings planned

- `src/styles/utilities.css` is 993 lines and mixes primitives, patterns,
  sections, routes, and responsive overrides in one global cascade.
- `LandingPage` and `SectionLab` each maintain their own section list, so the
  preview route can drift from the production page.
- Demo controls expose selection only through `data-*`, mobile layer buttons
  can lose accessible names, FAQ close actions are tracked as opens, and demo
  content arrays are typed as ordinary arrays despite indexed dereferences.
- Verification exists as separate scripts but not as one enforced gate, and the
  starter has no contributor or architecture documentation.
- Vite tooling is listed as production dependencies, and the TypeScript caret
  range can move past the `typescript-eslint` peer window.

## Findings considered and deferred

- Primitive-level analytics in `Button`: valid architecture concern, but the
  current typed `analyticsEvent` surface is small and localized. Defer until a
  tracked-control wrapper or instrumentation policy exists.
- Direct section imports of `landingPage`: important for localization/CMS work,
  but the section registry plan is the safer first step. Revisit when variants
  or CMS adapters are real.
- Unused `src/lib/motion.ts`: stale surface, but low risk. Delete or wire it in
  during a motion-system pass after CSS ownership is clarified.
- Lazy-loading `/sections`: likely useful once the workbench grows, but bundle
  impact was not measured because the improve audit avoided build artifacts.
- Replacing `lucide-react`: install size is non-trivial, but runtime impact was
  not measured. Keep until icon breadth or bundle data justifies a swap.
